# Benchmarks

This directory holds reproducible causet inputs for `cones.py`.

The broader research direction is described in [`research_agenda_2026.md`](/home/adnac/sorkin/research_agenda_2026.md).

## `tesis_like_6.in`

- Generated by sprinkling `6` events in a 1+1 Minkowski causal diamond.
- Seed: `1987`
- Space dimension passed to `cones.py`: `2`

Why this one:

- It is small enough that the annealer often reaches energy `0` or very close to it.
- It gives a usable benchmark for checking that the port actually finds a good embedding.
- It is the best current candidate for comparing schedule behavior against the thesis-style optimization.

Run it with:

```bash
python3 cones.py benchmarks/tesis_like_6.in --dim 2 --output runs/tesis_like_6.out --plot runs/tesis_like_6.svg
```

For batch comparisons:

```bash
python3 cones.py benchmarks/tesis_like_6.in --dim 2 --sweep 12 --sweep-dir runs/tesis_like_6_sweep
python3 analyze_sweep.py sweep_report.csv --report-md sweep_analysis.md --plot-svg sweep_analysis.svg
```

## Schedule sweeps

To compare cooling schedules directly on a fixed benchmark:

```bash
python3 schedule_sweep.py benchmarks/tesis_like_12.in --dim 2 \
  --seed-count 4 --temp-count 2 --cooling-count 3 \
  --run-csv runs/schedule_runs.csv \
  --summary-csv runs/schedule_summary.csv \
  --report-md runs/schedule.md \
  --heatmap-svg runs/schedule.svg
```

The thesis defaults are preserved in `cones.py`:

- initial temperature: `100.0`
- cooling factor: `0.9`
- acceptance scale: `4.0`
- warmup and anneal block limits: `100`

On `tesis_like_12.in`, a direct control run with those defaults gave:

- mean final energy: `20.021376`
- best seed in the tested set: `1962`
- zero rate: `0.00`

In the refined schedule sweep around the empirical optimum, the best schedule found was:

- initial temperature: `180.0`
- cooling factor: `0.8`
- mean final energy: `0.166158`
- zero rate: `0.00`

That is a large gap, so the historical schedule is faithful to the thesis but not optimal for this particular benchmark.
The mean final energy is lower by about `99.17%` relative to the thesis-default control run on the same benchmark.

## Mass Exploration

For larger-scale exploration beyond the historical single-run workflow, use [`ensemble_scan.py`](/home/adnac/sorkin/ensemble_scan.py).

It runs many seeds across a schedule grid, then summarizes:

- success rate
- final-energy statistics
- best seed per schedule cell
- a heatmap SVG for phase-like behavior

Recommended smoke command:

```bash
python3 ensemble_scan.py benchmarks/tesis_like_6.in --dim 2 --gpu-first --backend auto \
  --seed-start 1959 --seed-count 2 \
  --temp-min 100 --temp-max 180 --temp-count 2 \
  --cooling-min 0.8 --cooling-max 0.9 --cooling-count 2 \
  --workers 1 \
  --run-csv /tmp/ensemble_runs.csv \
  --summary-csv /tmp/ensemble_summary.csv \
  --report-md /tmp/ensemble.md \
  --heatmap-svg /tmp/ensemble.svg
```

The same workflow is also exposed as `make ensemble`.
That target prefers the shared CUDA backend when a GPU is available and falls back to CPU when it is not.

## Phase Diagram

The first research milestone is implemented by [`phase_diagram.py`](/home/adnac/sorkin/phase_diagram.py).
It generates sprinkled causets for multiple `n` values, scans target dimensions, and reports success probability plus median final energy.

```bash
python3 phase_diagram.py --n-values 6,12 --dim-min 1 --dim-max 4 \
  --seed-start 1959 --seed-count 4 \
  --gpu-first --backend auto \
  --runs-csv /tmp/phase_runs.csv \
  --summary-csv /tmp/phase_summary.csv \
  --report-md /tmp/phase.md \
  --heatmap-svg /tmp/phase.svg
```

The same smoke workflow is exposed as `make phase`.
`phase_diagram.py` prints progress by default. Use `--quiet` only when running it from another script.
It also writes partial outputs after each completed cell by default, so an interrupted long run still leaves usable CSV, Markdown, and SVG files.

For frontier scans, use the adaptive fast mode:

```bash
python3 phase_diagram.py --n-values 6,12,16,24 \
  --dim-min 1 --dim-max 4 \
  --seed-start 1959 --seed-count 8 \
  --gpu-first --backend auto \
  --fast-frontier \
  --partial cell \
  --runs-csv /tmp/phase_frontier_runs.csv \
  --summary-csv /tmp/phase_frontier_summary.csv \
  --report-md /tmp/phase_frontier.md \
  --heatmap-svg /tmp/phase_frontier.svg
```

`--fast-frontier` reduces the annealing budget, sets a per-run timeout, and stops a cell early when it is dominated by timeouts or clearly high energies.

## Two-Phase Frontier Protocol

The planned workflow is now explicit:

1. Use `phase_diagram.py --fast-frontier` as a triage pass.
2. Use [`phase_refine.py`](/home/adnac/sorkin/phase_refine.py) to spend deeper annealing only on selected cells.

Automatic refinement from a fast summary:

```bash
python3 phase_refine.py \
  --source-summary-csv /tmp/phase_frontier_summary.csv \
  --seed-start 1959 --seed-count 16 \
  --gpu-first --backend auto \
  --runs-csv /tmp/phase_refine_runs.csv \
  --summary-csv /tmp/phase_refine_summary.csv \
  --report-md /tmp/phase_refine.md \
  --heatmap-svg /tmp/phase_refine.svg
```

Manual refinement of the cells currently worth spending time on:

```bash
python3 phase_refine.py --cells 6:3,6:4,12:2,16:2 \
  --seed-start 1959 --seed-count 16 \
  --gpu-first --backend auto \
  --partial cell \
  --runs-csv /tmp/phase_refine_runs.csv \
  --summary-csv /tmp/phase_refine_summary.csv \
  --report-md /tmp/phase_refine.md \
  --heatmap-svg /tmp/phase_refine.svg
```

The same manual refinement is exposed as `make refine`.

## Refined Frontier Result

The first manual second-phase run used:

- cells: `6:3,6:4,12:2,16:2`
- annealing seeds: `1959..1974`
- runs per cell: `16`
- backend: `auto` with GPU preference

Summary:

| n | dim | runs | success_rate | median_final | min_final | interpretation |
| ---: | ---: | ---: | ---: | ---: | ---: | --- |
| 6 | 3 | 16 | 0.125 | 0.037875 | 0.000000 | embeddable, but seed-sensitive |
| 6 | 4 | 16 | 0.500 | 0.000010 | 0.000000 | cleanest solved cell |
| 12 | 2 | 16 | 0.0625 | 0.057112 | 0.000000 | embeddable, mostly algorithm-limited |
| 16 | 2 | 16 | 0.0625 | 0.406879 | 0.000001 | current useful frontier |

Interpretation:

- `n=6, dim=4` is the current clean success case.
- `n=12, dim=2` reaches exact zero in at least one seed, so failures here are not evidence of non-embeddability.
- `n=16, dim=2` reaches a near-exact solution in one seed, but the success probability is low. This points to a difficult energy landscape rather than an obviously impossible embedding.
- `n=24` should remain out of the brute-force queue until the search algorithm improves.
